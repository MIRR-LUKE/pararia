# 国内エンタープライズ向けプライバシー / コンプライアンスパック

最終更新: 2026-04-25

## 1. 目的

この文書は、PARARIA を日本国内の上場企業、学校法人、学習塾、教育関連事業者に導入する際の、プライバシー・セキュリティ・コンプライアンス審査に回答するための実務パックである。

PARARIA は「海外展開を前提にしたサービス」ではなく、**日本向けの教育・面談業務 SaaS として提供しながら、個人情報保護、監査証跡、バックアップ、委託先管理、越境移転説明について世界水準の運用品質を満たす**ことを前提にする。

この文書は法的助言ではない。契約書、個人情報取扱規程、委託契約、学校法人ごとの内部規程との整合は、導入先と当社双方の法務・情報システム・個人情報保護管理者が確認する。

## 2. 審査回答の基本方針

### 2.1 国内向けであること

- 主たる提供対象は日本国内の教育・保護者対応・面談記録業務である。
- 画面、運用、契約、請求、サポート、障害連絡は日本語を標準とする。
- 海外拠点展開やグローバル販売ではなく、日本国内の学校法人・企業審査に耐える説明責任を優先する。
- 利用するクラウド、生成 AI、監視、メール等の一部サービスに国外事業者が含まれる場合は、委託先・データ所在地・越境移転の説明を明示する。

### 2.2 世界水準として守ること

- 個人データの収集目的、利用目的、保存期間、削除方法を文書化する。
- 音声、文字起こし、生徒情報、保護者向けレポート、監査ログのライフサイクルを分けて管理する。
- 本番データの復旧は DB だけでなく runtime object も含めて設計する。
- 監査ログ、バックアップ結果、復旧訓練、委託先レビュー、権限棚卸しを証跡として残す。
- 決済ゲートウェイは本パックの対象外とし、直販、契約、請求書払い、SLA、オンボード、オフボード、停止、更新を中心に管理する。

## 3. 対象データ

| 分類 | 例 | 主なリスク | 管理方針 |
| --- | --- | --- | --- |
| 生徒・保護者・担当者情報 | 氏名、所属、連絡先、対応履歴 | 個人情報の目的外利用、誤閲覧 | 組織単位でアクセス制御し、契約目的外に利用しない |
| 面談音声 | 録音ファイル、live chunk、manifest | 要配慮に近い会話内容、漏えい | 短期保存を前提にし、保存期間経過後 hard delete |
| 文字起こし | raw transcript、reviewed text、segments | 誤記録、機微情報の残存 | レビュー導線と削除期間を分離し、生データは長期保存しない |
| structured artifact | `ConversationLog.artifactJson` | 正本の改変、根拠不明な生成 | 画面表示・レポート生成の正本として扱う |
| 保護者向けレポート | report body、delivery event | 誤送信、説明責任不足 | 共有履歴と delivery event を証跡化する |
| 監査ログ | ログイン、作成、更新、削除、配信操作 | 不正操作の追跡不能 | アプリ画面や通常 API から削除しない |
| バックアップ | DB dump、Blob backup、manifest | 復旧不能、過剰保持 | 復旧目的に限定し、アクセスを分離する |

保存期間と削除方式の正本は [data-retention-policy.md](./data-retention-policy.md) を参照する。バックアップ・復旧の正本は [db-backup-recovery.md](./db-backup-recovery.md) を参照する。

## 4. 個人情報保護法・学校法人審査での説明項目

### 4.1 利用目的

- 面談記録、学習支援、保護者報告、内部引き継ぎ、対応品質向上のために利用する。
- 導入先から委託された業務の範囲外で、生徒・保護者データを広告配信、第三者販売、別サービスの名寄せに利用しない。
- 生成 AI を利用する場合も、入力・出力は導入先業務の処理目的に限定する。

### 4.2 取得・入力

- 利用者が管理画面、録音、アップロード、編集、レポート生成操作を通じて入力する。
- 導入前に、録音対象者への説明、同意取得、校内・社内規程との整合を導入先が確認する。
- 録音禁止区域、同意が取れない面談、特に機微性の高い相談は、導入先の運用判断で利用を制限する。

### 4.3 第三者提供と委託

- PARARIA は導入先から受託した処理者としてデータを扱う。
- サービス提供に必要なクラウド、データベース、ストレージ、メール、監視、生成 AI 等は再委託先として管理する。
- 委託先・所在地・越境移転の説明は [data-location-and-subprocessors.md](./data-location-and-subprocessors.md) に集約する。
- 法令に基づく開示要請を除き、導入先の指示なく第三者へ個人データを提供しない。

### 4.4 安全管理措置

- 組織的安全管理措置: 管理責任者、権限棚卸し、委託先レビュー、インシデント報告経路を定める。
- 人的安全管理措置: 運用担当者に守秘義務、アクセス最小化、持ち出し禁止を課す。
- 物理的安全管理措置: 本番データは管理されたクラウド環境に置き、個人端末への恒常保存を禁止する。
- 技術的安全管理措置: 認証、認可、監査ログ、環境変数分離、バックアップ、復旧訓練を実施する。

## 5. アクセス制御

### 5.1 権限原則

- 最小権限を原則とし、役割ごとに閲覧・編集・配信・管理権限を分ける。
- 本番 DB、Blob、バックアップ、監視ログへのアクセスは、アプリ利用権限とは別管理にする。
- 退職、異動、委託終了、端末紛失時は、管理者がアカウントとセッションを停止する。

### 5.2 定期棚卸し

- 月次: 管理者アカウント、外部委託者、サポート用一時権限を確認する。
- 四半期: 本番環境、バックアップ、クラウド管理画面、GitHub Actions secrets のアクセス権を確認する。
- 年次: 導入先ごとの契約・委託範囲・保存期間・SLA を再確認する。

### 5.3 証跡

棚卸しの証跡には次を残す。

- 実施日
- 実施者
- 対象システム
- 追加・削除・継続したアカウント
- 例外承認者
- 次回対応期限

## 6. ログ・監査

### 6.1 監査ログ対象

- ログイン、ログアウト、認証失敗
- 生徒、面談、会話ログ、レポートの作成・更新・削除・アーカイブ
- 音声アップロード、文字起こし、生成、再生成
- 保護者向けレポートの共有・送信・取り消し
- 管理者設定、権限変更、組織設定変更
- バックアップ、復旧、削除 batch、手動メンテナンス

### 6.2 監査ログの扱い

- 監査ログはアプリ画面や通常 API から削除しない。
- 個人データ削除時も、不正調査・説明責任に必要な最小限の操作証跡は保持する。
- ログには本文や音声そのものを過剰に保存せず、操作種別、対象 ID、実施者、時刻、結果を中心に残す。

## 7. インシデント対応

### 7.1 重大度

| 重大度 | 例 | 初動 |
| --- | --- | --- |
| Sev1 | 個人データ漏えい、広範囲の誤送信、本番 DB 破損 | 書き込み停止、影響範囲特定、導入先へ速報 |
| Sev2 | 一部組織の閲覧障害、限定的な誤生成、復旧可能なデータ欠損 | 原因切り分け、回避策提示、復旧計画共有 |
| Sev3 | 軽微な表示不具合、限定的な遅延 | 通常サポートで対応 |

### 7.2 初動手順

1. 事象、検知時刻、影響範囲、暫定原因を記録する。
2. 必要に応じて該当機能、該当組織、該当アカウントの利用を一時停止する。
3. 監査ログ、アプリログ、バックアップ、配信履歴を保全する。
4. 導入先窓口へ速報を出し、続報時刻を明記する。
5. 恒久対応、再発防止、削除・訂正・通知要否を判断する。

## 9. 審査回答チェックリスト

- 日本国内向けサービスであることを説明したか
- 世界水準の安全管理・監査・復旧運用を採用することを説明したか
- 決済ゲートウェイを利用せず、直販・請求書払いで契約管理することを説明したか
- 個人データの利用目的、保存期間、削除方法を説明したか
- 音声・文字起こし・artifact・レポート・監査ログを分けて説明したか
- 委託先、所在地、越境移転を説明したか
- オンボード、オフボード、停止、更新、契約終了後削除を説明したか
- 障害、漏えい、誤送信時の連絡経路と証跡保全を説明したか
