# 保護者レポート再設計 提案書

更新日: 2026-04-13

## 0. 結論

今回の保護者レポートは、単に「長文を短くする」では足りません。目指すべき成果物は、

- 保護者が30秒以内に読み切れる
- 「この子は見てもらえている」「塾は次の一手まで持っている」と伝わる
- 共同子育てを強く要求せず、「この領域は塾に任せてよさそう」と思わせる

という、**短いが軽くない進捗報告**です。

そのための基本方針は次の3つです。

1. **主報告は短くする**
2. **内面的な話は残すが、学習・進路との因果が見えるものだけ残す**
3. **詳細は捨てず、主報告から切り離して段階的に見せる**

要するに、最適解は「情報量を減らす」ではなく、**情報の階層を作ること**です。

---

## 1. クライアントFBの要約

今回の議論を整理すると、論点は以下でした。

- りんゆーさん:
  - いまのレポートは少し長い
  - 通常塾の保護者は長文を丁寧に読まない
  - ただし、雑な報告では信頼は生まれない
  - 詳細な共同子育てへのナーチャリングより、「ここに任せておけばよさそう」と思わせる方がビジネス的に強い
- 浅見さん:
  - 情報が浅い場合は、親が納得するために説明量が必要
  - 計画と進捗が理路整然としていることが重要
- ゆっかさん:
  - APSの保護者は、子どもの内面に関心がないわけではない
  - ただ本質的な不安は「GMARCH級以上に届くのか」「何も変化が見えないのではないか」
  - 内面的・精神的な話は、学習モチベーションや進路不安と結びついているときに価値になる
  - 長すぎず、読み切れることは重要

ここから導ける共通認識は以下です。

- 保護者の不安は「計画があるか」だけではなく、**本人に変化が出ているか**にある
- ただし、感情面だけでも足りず、**学習・進路にどうつながるか**が必要
- したがって保護者レポートは、**進捗報告 + 変化の意味づけ + 塾の主導感**を短く伝えるべき

## 2. 目指すべきUX/UIの北極星

### 保護者向け

保護者が読了後、30秒以内に次の4つを理解できることを北極星に置きます。

1. **うちの子はちゃんと見られている**
2. **今月、何かしらの変化が起きている or 起きていない理由が見えている**
3. **塾は次に何をするか決めている**
4. **家庭が今すぐ大きく動かなくてもよい / 動くなら最小限でよい**

### 塾オペレーター向け

講師・室長は、送付前確認で次の3つを一目で判断できるべきです。

1. このレポートは長すぎないか
2. このレポートは「変化」と「次の一手」が入っているか
3. 内面的記述が、学習や進路の話とつながっているか

### 事業向け

レポートの役割は、保護者を高関与化することではなく、

- 離脱不安を減らす
- スイッチング理由を減らす
- 「ここは見てくれている」という委任感を作る

ことです。

## 3. リサーチ要点

### 3-1. Plain language と情報階層は必須

- DfE の Plain Language standard は、良いコンテンツの条件を「**find / understand / act**」と定義し、短い文、明確な見出し、必要な情報を必要な場所で出すことを求めています。
- ONS の structuring content guidance では、**Web上の平均読者は本文の20%〜28%しか読まない**としたうえで、結論先出し、短い段落、短い文、明確な見出しを推奨しています。
- ONS の details component は、詳細情報を折りたたんで二次情報として見せる設計を推奨しています。

**示唆**

- 主報告は短く、詳細は後ろに隠すべき
- 文章の品質は「たくさん書けているか」ではなく、「見つけやすく、理解しやすく、行動しやすいか」で判断すべき

### 3-2. 学校-家庭コミュニケーションは「短くても個別具体」だと強い

- Matthew Kraft の整理では、teacher-parent communication は依然として少ない一方、**頻度が高く、個別化された連絡は学習成果に効く**とされています。
- 同記事では、**先生からの weekly one-sentence individualized messages が単位取得失敗率を41%下げた**事例が紹介されています。

**示唆**

- 長文の包括レポートでなくても、個別性と具体性があれば価値は高い
- 重要なのは文字数より、**「自分の子のこととして書かれているか」**です

### 3-3. 強い学校は「一対一」「学習に anchored」「しつこく切れない」連絡をしている

- Learning Heroes の 2026 年 family-school partnership study では、成果の高い Bright Spot 校の特徴として、
  - one-on-one
  - proactive
  - academically focused
  - personal and enduring
  なコミュニケーションが挙げられています。
- 同レポートでは、強い学校は学校全体のお知らせよりも、**個別の学習進捗と目標**に紐づく連絡を重視していました。

**示唆**

- 保護者レポートに必要なのは、学校全体の説明や一般論ではなく、**本人の今月の学習的な現在地**です
- ただの進捗報告ではなく、**個別の関係性が見える一文**が信頼を押し上げます

### 3-4. 内面的な記述は有効。ただし「学習にリンク」していることが前提

- Harvard GSE の family engagement 実践知では、強い連携は positive communication と learning-focused communication から始まると整理されています。
- 同記事は、会話を rules や behavior だけでなく、**academic content and strategies** につなげることを推奨しています。

**示唆**

- 友人関係や気分の記述をそのまま書くのは危険
- ただし「周囲の受験意識に引っ張られて不安が強まった」「自己効力感の変動が勉強の着手に影響している」のように、**モチベーションや進路判断に結びついているなら価値が高い**

## 4. 現状診断

## 4-1. いまの実装の良い点

現状実装には、既に良い土台があります。

- 使うログを明示的に選ぶ
- `structuredArtifact` を優先して根拠を取る
- bundle quality で「強い材料」「弱い材料」「次回確認」「保護者共有ポイント」を抽出している
- 送付前チェックや共有履歴も見える

つまり、**何を残すべきかを考える材料そのものは既に揃っている**状態です。

## 4-2. いまの実装の詰まりどころ

一方で、現状は「信頼されるために十分に書く」方向にかなり強く最適化されています。

### 現状の構造的な長文化要因

- 親向け本文が **7セクション固定**
  - 今回の様子
  - 学習状況の変化
  - 講師としての見立て
  - 科目別またはテーマ別の具体策
  - リスクとその意味
  - 次回までの方針
  - ご家庭で見てほしいこと
- 品質評価で **本文が短いと失敗扱い**
  - `report_is_too_short` の閾値が 650 文字未満
- repair prompt でも
  - 7セクション維持
  - generic 段落の削減
  - 具体性追加
  を求めており、**短く書くインセンティブがほぼない**

### 現状UXの課題

- 生成前は bundle quality が見えるが、**最終的に親が受け取る短い要約像**が見えない
- 生成後は paragraph stack として表示され、**「親が最初に読むべき核」がUI上で前景化されていない**
- 「標準版」と「詳細版」の区別がなく、**全ケースでフルセットの説明に寄る**

### まとめ

いまの問題は文章力ではなく、**情報設計そのものが completeness 側に寄っていること**です。

## 5. これから作るべき成果物

提案するのは、従来の「保護者レポート」ではなく、次の設計です。

## 5-1. 新しい主成果物: Parent Progress Brief

主成果物は、**短い進捗ブリーフ**に変えます。

### 標準フォーマット

- 4ブロック構成
  - 今月の状況
  - 見えている変化
  - 次回までに塾がやること
  - ご家庭では（必要なときだけ）
- 文字量目安
  - 標準: **180〜320字**
  - 高リスク / 大きな方針変更あり: **450〜700字**
- 読了時間
  - 標準: **20〜35秒**

### 伝えるべき中身

必ず入れる

1. **現在地**
2. **変化 or 変化がまだ出ていないことの意味**
3. **塾の次アクション**

条件付きで入れる

4. **内面的な観察**
5. **家庭での一言確認**
6. **リスク**

### 入れ方のルール

- 内面的記述は **1つまで**
- しかも、**勉強の着手・継続・進路判断に結びつくものだけ**
- 家庭への依頼は **1つまで**
- 家庭での行動コストは低くする
- 「講師の頑張り」ではなく、「子どもの状態」と「塾の制御感」を見せる

## 5-2. 詳細版は捨てずに二層化する

詳細は不要ではありません。不要なのは、**主報告に全部のせること**です。

したがって成果物は二層構造にします。

### Layer 1: 親向け主報告

- 短い
- 読み切れる
- 変化と方針が一目で分かる

### Layer 2: 詳細版

- 室長・講師の確認用
- 必要に応じて保護者にも見せられる
- 使ったログの根拠、詳しい補足、長めの説明を持つ

これは world-class UX でいう **progressive disclosure** に当たります。

## 6. 残す情報 / 削る情報のアルゴリズム

今回もっとも重要なのはここです。

## 6-1. 判断基準

各候補情報を、以下の6軸で採点します。

1. **保護者不安への関係度**
   - 進路不安
   - 勉強していない不安
   - 何も変化がない不安
2. **根拠強度**
   - ログ上で明確に確認できるか
3. **行動価値**
   - 次の塾アクションや家庭の見守りにつながるか
4. **変化性**
   - 前回からの変化、切り替わり、停滞が見えるか
5. **個別性**
   - その子特有の状態として読めるか
6. **冗長コスト**
   - 同じ意味の繰り返しではないか

## 6-2. 採用ルール

### 必ず残す

- 今月の現在地を表す事実 1つ
- 変化または停滞の意味づけ 1つ
- 塾の次アクション 1つ

### あれば残す

- モチベーションや感情の変化 1つ
  - ただし、学習行動か進路不安に直結する場合だけ
- 家庭での観察ポイント 1つ
  - 一言で済むものだけ

### 基本は削る / 詳細版に送る

- 面談の時系列説明
- 同じことの言い換え
- generic な励まし
- 学習に関係しないエピソード
- 講師の内部判断の長い背景説明
- 親が今読まなくても困らない補足

## 6-3. 実際の選定ロジック

### ステップ1

ログから候補文を抽出し、以下のタグを付ける。

- `state`
- `change`
- `academic_impact`
- `motivation_link`
- `school_action`
- `home_watch`
- `risk`

### ステップ2

各候補にスコアを付ける。

`priority = anxiety_relevance + evidence_strength + actionability + change_signal + individuality - verbosity_cost - duplication_cost`

### ステップ3

次の順で埋める。

1. `state` の最高スコア
2. `change` または `motivation_link` の最高スコア
3. `school_action` の最高スコア
4. `home_watch` が low burden なら追加

### ステップ4

以下の条件なら詳細版に切り替える。

- 大きな方針変更がある
- 受験直前 / 志望校判断が絡む
- high risk
- ログが浅く、短文だと誤読されやすい

### このアルゴリズムの意味

「全部をまんべんなく伝える」ではなく、
**親の不安を最も効率よく下げる因果線だけを残す**ということです。

## 7. Prompt tuning 提案

## 7-1. 今の prompt の問題

現状の prompt は、品質を上げる意図としては正しい一方で、

- 7セクションを維持する
- 具体化を強く促す
- 短すぎる出力をエラー扱いする

ため、**簡潔に良い報告を書くより、厚く破綻しない報告を書く方向**に寄っています。

これは今回のクライアント意図とはズレます。

## 7-2. 新しい prompt の思想

新 prompt は次の思想で作るべきです。

- あなたは包括レポートを書かない
- あなたの仕事は、保護者の不安を下げ、塾への委任感を高めること
- 30秒以内に要点が伝わることを優先する
- 一番強い因果線だけを残す
- 内面的情報は、学習・進路との関係があるときだけ使う
- 情報が薄いときは膨らませず、「次回何を見にいくか」を書く

## 7-3. 生成パイプラインの再設計

### 現状

- 1回で parent-facing full draft を生成
- 不十分なら detail を足す方向で repair

### 提案

2段階に分けます。

#### Step A: Evidence selection

モデルに最初にやらせるのは執筆ではなく、選定です。

出力例:

```json
{
  "mode": "standard",
  "state": "語句理解の曖昧さが国語の基礎問失点につながっている",
  "change": "本人も基礎で落とした自覚が出てきた",
  "motivation_link": "",
  "school_action": "次回は大問2の解き直しと語句理解の確認を優先する",
  "home_watch": "間違えた問題を解き直したかだけ確認してもらう",
  "risk": "基礎で失点が続くと得点が安定しない",
  "detail_candidates": [
    "語句の意味を後回しにしたまま解いていた",
    "難問対応より基礎精度の立て直しが先"
  ]
}
```

#### Step B: Parent-facing rendering

その後に、mode ごとの長さ制約で親向け文面を生成します。

### mode 定義

- `concise`
  - 120〜220字
  - 順調時
- `standard`
  - 180〜320字
  - もっとも多い通常ケース
- `detailed`
  - 450〜700字
  - 高リスク、方針変更、重大面談

## 7-4. Prompt ルール案

### System prompt の主旨

```text
あなたは学習塾の保護者向け progress brief を作成する。
包括レポートではなく、30秒で読み切れる短い報告を作ること。
目的は、保護者に「この子は見られている」「塾は次の一手を持っている」と感じてもらうこと。
共同子育てを強く求めず、家庭への依頼は最大1つにする。
内面的な記述は、学習行動・進路不安・モチベーションに結びつく場合に限って使う。
情報が薄い場合は膨らませず、次回何を確認するかを書く。
```

### Validation を coverage-based に変える

いまの `650文字未満は弱い` という判定をやめ、代わりに以下を満たしているかで判定します。

- 現在地がある
- 変化 or 停滞の意味がある
- 塾の次アクションがある
- 家庭向けの確認が必要なら1つある
- unsupported claim がない

### 重要な修正点

- `report_is_too_short` を削除 or mode別閾値に変更
- 7セクション固定をやめる
- repair prompt は「詳しくしろ」ではなく「核が欠けている箇所だけ補え」に変える
- `max_completion_tokens` も mode 別に落とす

- 809字 -> 231字
- 11段落 -> 4ブロック
- 「問題の核心」「変化」「次アクション」が前景化
- 親の行動負荷を最小化

### 何が残り、何が落ちたか

残したもの

- 失点の核心
- 本人の変化
- 塾の方針
- 家庭での確認点

落としたもの

- 似た意味の繰り返し
- 長い背景説明
- 7観点を全部独立させる構造

## 9. UX/UI 設計方針

## 9-1. 送付前確認画面

いまの bundle quality は良いので活かします。ただし、次を追加すると完成度が上がります。

- `想定モード`
  - concise / standard / detailed
- `想定文字数`
- `親が最初に読む1文`
- `長くなる理由`
  - 高リスク
  - 方針変更
  - 面談密度高

これにより、講師は生成前に「長くなる妥当性」を判断できます。

## 9-2. 生成後ドラフト画面

現在は paragraph stack に近い見え方です。これを以下に変えます。

### 上段

- 実際に親へ送る主報告を、そのまま表示
- ここが main preview

### 下段

- 折りたたみの詳細
  - 使ったログ
  - 追加説明
  - リスク補足

これで、講師も「親がどう読むか」を先に確認できます。

## 9-3. 履歴画面

履歴は日付と状態だけでなく、**一行サマリー**を持つべきです。

例:

- `基礎問の取りこぼしが主因。次回は国語の語句確認を優先`

これがあると、後から見返したときの認知負荷が下がります。

## 11. KPI と受け入れ条件

### KPI

- 標準レポート中央値: 180〜320字
- 送信前の手修正率: 現状比で低下
- 1レポートの確認時間: 現状比で短縮
- 「何も変化がないように見える」ケースの減少

### 受け入れ条件

- 主報告だけで current state / change / next action が分かる
- 7観点のフル説明を常態化しない
- 学習と無関係な内面描写を減らす
- ただし、高リスクケースでは詳細を出せる

---

## 12. 最終提案

この案件で作るべきものは、**長文レポートの圧縮版**ではありません。

作るべきものは、

- 不安を下げる
- 変化を見せる
- 塾の制御感を伝える
- 家庭の負担を増やしすぎない

という目的に最適化された、**短い progress brief + 詳細版の二層設計**です。

クライアントFBを踏まえると、最も筋が良いのは次の一文に集約されます。

> 保護者を共同運営者に育てるのではなく、  
> 「ここは塾がちゃんと見てくれている」と感じてもらうための報告にする。

今回の再設計は、この方向に最も強く寄せるべきです。
