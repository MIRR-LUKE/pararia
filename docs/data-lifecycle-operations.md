# データライフサイクル運用手順

最終更新: 2026-04-25

## 1. 目的

この文書は、PARARIA におけるデータの作成、利用、保存、削除、アーカイブ、復旧、証跡管理を、上場企業・学校法人の審査に耐える運用手順として定義する。

PARARIA は日本向け SaaS であり、海外展開ではなく、国内の教育・面談業務に必要な説明責任を満たすことを優先する。一方で、データ最小化、復旧可能性、監査証跡、委託先管理については世界水準の運用を採用する。

保存期間と削除方式の詳細は [data-retention-policy.md](./data-retention-policy.md) を正本とする。DB と runtime object のバックアップ・復旧手順は [db-backup-recovery.md](./db-backup-recovery.md) を正本とする。

## 2. ライフサイクル全体像

| 段階 | 主な操作 | 証跡 | 関連文書 |
| --- | --- | --- | --- |
| 取得 | 録音、入力、アップロード、連携 | 操作ログ、作成者、作成時刻 | 本文書 |
| 処理 | 文字起こし、整形、生成、レビュー | job log、生成結果、レビュー履歴 | 本文書 |
| 利用 | 画面表示、レポート作成、共有 | 閲覧・更新・配信ログ | 本文書 |
| 保持 | artifact、summary、report、audit | 保存期間設定、バックアップ結果 | data-retention-policy |
| 削除 | retention cleanup、個別削除、契約終了削除 | 削除 batch log、承認記録 | data-retention-policy |
| 復旧 | archive restore、DB restore、Blob restore | 復旧実行記録、検証結果 | db-backup-recovery |

## 3. データ種別ごとの運用

### 3.1 生徒・保護者・担当者情報

- 導入先組織ごとに論理的に分離して扱う。
- 生徒の通常削除は hard delete ではなく archive を優先する。
- archive 前 snapshot は `StudentArchiveSnapshot` に保存する。
- 誤操作時は [db-backup-recovery.md](./db-backup-recovery.md) の archived student restore 手順を使う。
- 契約終了時は、導入先と合意したエクスポート期間後に削除対象とする。

### 3.2 面談音声・live chunk・manifest

- 音声は永続保存を前提にしない。
- 保存期間は `PARARIA_AUDIO_RETENTION_DAYS` に従う。
- cleanup 実行時に runtime storage から物理削除する。
- `SessionPart.storageUrl` は cleanup 後に null へ更新する。
- バックアップは復旧目的に限定し、通常の閲覧や分析用途には使わない。

### 3.3 生文字起こし

- `rawTextOriginal`、`rawTextCleaned`、`reviewedText`、`rawSegments` は、生成・レビュー・検証のための中間データとして扱う。
- 保存期間は `PARARIA_TRANSCRIPT_RETENTION_DAYS` に従う。
- cleanup 実行時に hard delete する。
- 固有名詞候補など、文字起こしから派生した補助データも同時に削除する。

### 3.4 structured artifact

- 画面表示、レポート生成、説明資料の正本は `ConversationLog.artifactJson` とする。
- `summaryMarkdown` は artifact から生成される派生物として扱う。
- 個別削除または契約終了削除までは保持できる。
- 保護者レポート参照中の artifact を削除する場合は、source trace から外す。

### 3.5 保護者向けレポートと共有履歴

- レポート本文、共有履歴、delivery event を紐づけて管理する。
- レポート削除時は delivery event と一緒に削除する。
- `ReportDeliveryEvent` 単体の cleanup は `PARARIA_REPORT_DELIVERY_EVENT_RETENTION_DAYS` に従う。
- 誤送信・再送・取り消しがあった場合は、対象、時刻、担当者、理由を証跡化する。

### 3.6 監査ログ

- 監査ログはアプリから削除しない。
- 通常の個別削除や契約終了削除でも、必要最小限の操作証跡は保持する。
- 本文や音声そのものを監査ログへ過剰に複製しない。

## 4. 日次運用

### 4.1 cleanup 確認

日次で確認する項目:

- audio retention cleanup の実行有無
- transcript retention cleanup の実行有無
- report delivery event cleanup の実行有無
- 失敗 job の件数
- cleanup 後に参照切れの画面エラーがないこと

### 4.2 バックアップ確認

日次で確認する項目:

- DB backup 成功
- Blob backup または manifest backup 成功
- GitHub Actions artifact の作成
- Supabase backup / PITR 状態
- バックアップ失敗時の再実行記録

詳細は [db-backup-recovery.md](./db-backup-recovery.md) を参照する。

### 4.3 監査ログ確認

日次で確認する項目:

- 管理者権限の追加・削除
- 大量削除、大量エクスポート、大量配信
- 失敗した認証試行の急増
- 本番環境への手動アクセス
- backup / restore / migration 操作

## 5. 週次運用

- 任意の DB dump を検証用 DB に restore する。
- Blob backup から任意の音声を 1 件取り出して読めることを確認する。
- archived student を 1 件 restore し、画面で整合性を確認する。
- cleanup 対象件数と失敗件数を確認する。
- サポート問い合わせのうち、個人情報・削除・訂正・誤送信に関するものをレビューする。

## 6. 月次運用

- 権限棚卸しを実施する。
- 委託先、クラウド設定、環境変数、GitHub Actions secrets を確認する。
- Supabase PITR から sandbox project へ restore drill を実施する。
- データ保持期間と導入先契約の矛盾がないか確認する。
- SLA レポート、障害件数、対応時間、復旧訓練結果をまとめる。

## 7. 契約開始時のオンボード

### 7.1 事前確認

- 導入先の管理者、情報システム、法務、個人情報保護担当を確認する。
- 利用目的、録音対象、保護者説明、校内・社内規程との整合を確認する。
- データ所在地、委託先、越境移転説明を提供する。
- SLA、サポート時間、障害連絡先、請求書払い条件を確認する。
- 初期ユーザー、管理者、組織設定、権限ロールを確定する。

### 7.2 初期設定

- 組織 tenant を作成する。
- 管理者アカウントを発行する。
- 初期権限を最小限に設定する。
- 必要に応じて生徒・担当者データをインポートする。
- インポート元、件数、実施者、日時、検証結果を記録する。

### 7.3 初回運用確認

- 録音、文字起こし、レビュー、レポート作成、共有、削除の一連の操作を検証する。
- 誤送信防止、権限不足時の表示、アーカイブ復旧手順を確認する。
- 初回 2 週間は問い合わせと操作ログを重点確認する。

## 8. 契約終了時のオフボード

### 8.1 受付

- 契約終了日、最終利用日、データエクスポート要否を確認する。
- 導入先の承認者、受領者、削除承認者を確認する。
- 請求未払い、契約更新交渉中、法令保存義務の有無を確認する。

### 8.2 エクスポート

- エクスポート対象を合意する。
- 生徒、面談 artifact、レポート、共有履歴、必要な監査証跡を分けて出力する。
- 音声・生文字起こしは保存期間内に存在する場合のみ対象に含める。
- エクスポートファイルの作成者、保存場所、受領者、受領日時を記録する。

### 8.3 削除

- 最終利用日以降、管理者以外のログインを停止する。
- 合意した猶予期間後に tenant データを削除する。
- バックアップ内のデータは即時個別削除ではなく、バックアップ保持期間の満了により消去されることを説明する。
- 削除完了証跡には、対象組織、対象データ、実施者、実施日時、例外、承認者を残す。

## 9. 停止・凍結

### 9.1 停止条件

- 契約期間満了
- 重大な利用規約違反
- セキュリティ事故の封じ込め
- 長期未払い
- 導入先からの停止依頼

### 9.2 停止時の扱い

- 原則としてログインと新規作成を停止し、証跡と復旧可能性を維持する。
- 未払い停止では、ただちにデータ削除しない。
- セキュリティ事故では、該当アカウント、該当組織、該当機能のいずれか最小範囲で停止する。
- 停止理由、停止範囲、開始時刻、解除条件を記録する。

## 10. 証跡台帳

次の台帳を残す。

| 台帳 | 頻度 | 保管内容 |
| --- | --- | --- |
| backup 実行台帳 | 日次 | 成否、対象、artifact、再実行 |
| restore drill 台帳 | 週次・月次 | 復旧元、復旧先、検証結果 |
| cleanup 台帳 | 日次 | 対象件数、削除件数、失敗件数 |
| 権限棚卸し台帳 | 月次 | 対象アカウント、変更、承認 |
| 委託先レビュー台帳 | 四半期 | 委託先、用途、所在地、変更有無 |
| インシデント台帳 | 随時 | 事象、影響、対応、再発防止 |
| オフボード台帳 | 契約終了時 | エクスポート、停止、削除、例外 |

## 11. 例外管理

保存期間延長、削除保留、権限一時付与、本番データ調査などの例外は、次を満たす場合のみ許可する。

- 目的が明確である。
- 対象データと期間が限定されている。
- 承認者が記録されている。
- 終了条件が定義されている。
- 終了後に削除・権限剥奪・結果確認が行われている。

## 12. 完了判定

データライフサイクル運用は、次を満たした状態を完了とする。

- 保存期間と削除方法が [data-retention-policy.md](./data-retention-policy.md) と矛盾していない。
- 復旧手順が [db-backup-recovery.md](./db-backup-recovery.md) と矛盾していない。
- 日次・週次・月次の運用証跡が残っている。
- 導入先へ説明可能なオンボード・オフボード手順がある。
- 停止、更新、契約終了、削除の責任分界が契約文書と一致している。
