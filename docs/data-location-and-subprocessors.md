# データ所在地・委託先・越境移転説明

最終更新: 2026-04-25

## 1. 目的

この文書は、PARARIA のデータ所在地、委託先、再委託先、越境移転に関する説明を、日本国内の上場企業・学校法人の審査向けに整理する。

PARARIA は海外展開を前提にしたサービスではなく、日本国内の教育・面談業務向け SaaS として提供する。ただし、クラウド、生成 AI、監視、メール、開発運用の一部には国外事業者または国外リージョンが関与する可能性があるため、**日本向けだが世界水準の透明性で、データの所在、委託先、越境移転、変更通知、証跡管理を説明する**。

## 2. 基本方針

- 顧客データの主たる利用目的は、導入先から委託された教育・面談・保護者報告業務の処理である。
- 委託先はサービス提供に必要な範囲に限定する。
- 委託先ごとに、用途、扱うデータ種別、所在地、越境移転可能性、管理責任者を台帳化する。
- 委託先変更時は、契約または個別合意に従って導入先へ通知する。
- 決済ゲートウェイは対象外とし、直販・請求書払いに必要な事務情報のみを扱う。

## 3. データ所在地の考え方

### 3.1 主データ

主データには次を含む。

- 生徒・保護者・担当者情報
- 面談 session
- conversation
- structured artifact
- 保護者向けレポート
- delivery event
- settings
- audit

主データは、利用しているクラウド・DB・ストレージのリージョン設定に従って保存される。国内リージョンを利用できる場合は、日本国内リージョンを優先する。国内リージョンを利用できない機能または外部 API については、委託先台帳に所在地と越境移転の可能性を明記する。

### 3.2 runtime object

runtime object には次を含む。

- 録音ファイル
- live chunk
- manifest
- 生成途中の一時ファイル
- Blob backup

runtime object は DB backup だけでは復旧できないため、[db-backup-recovery.md](./db-backup-recovery.md) に従って DB backup と Blob backup を分けて管理する。

### 3.3 バックアップ

バックアップには次を含む。

- Supabase PITR / managed backups
- `pg_dump` による論理バックアップ
- Blob object backup
- GitHub Actions artifact

バックアップは復旧目的に限定し、通常の閲覧・分析・二次利用には使わない。バックアップ保持期間、復旧訓練、削除満了の扱いは [db-backup-recovery.md](./db-backup-recovery.md) を参照する。

## 4. 委託先台帳

実運用では、次の形式で委託先台帳を維持する。

| 区分 | 委託先 | 用途 | データ種別 | 所在地・リージョン | 越境移転可能性 | 備考 |
| --- | --- | --- | --- | --- | --- | --- |
| DB | Supabase | Postgres、認証補助、PITR | 主データ、監査ログ | 契約環境の設定に従う | あり得る | PITR / managed backups を含む |
| Object Storage | Vercel Blob | 音声、runtime object、Blob backup | 音声、live chunk、manifest | 契約環境の設定に従う | あり得る | DB backup とは別に保全 |
| Hosting | Vercel | Web app、API、Functions | リクエスト、ログ、一時処理 | 契約環境の設定に従う | あり得る | runtime logs を含む |
| 生成 AI / STT | 利用中の AI / STT provider | 文字起こし、生成、要約 | 音声、文字起こし、prompt、出力 | provider の設定に従う | あり得る | 学習利用の有無を契約設定で確認 |
| メール / 通知 | 利用中の mail provider | 通知、請求・サポート連絡 | 担当者連絡先、通知本文 | provider の設定に従う | あり得る | 保護者宛送信は運用設計に従う |
| 監視 / ログ | 利用中の monitoring provider | 障害検知、性能監視 | technical log、trace | provider の設定に従う | あり得る | 本文・音声の過剰送信を避ける |
| 開発運用 | GitHub / GitHub Actions | CI、backup workflow、artifact | secret 名、backup artifact | provider の設定に従う | あり得る | secret 管理と artifact 保持に注意 |
| 請求事務 | 社内請求管理 | 請求書払い、契約台帳 | 法人名、担当者、請求情報 | 日本国内運用を基本 | 原則なし | 決済ゲートウェイは使わない |

委託先名、リージョン、契約状態、学習利用設定、ログ保持期間は、導入環境ごとに最新値を確認して台帳に反映する。

## 5. 越境移転説明

### 5.1 説明の基本文

PARARIA は日本国内の教育・面談業務向け SaaS として提供されます。サービス提供にあたり、クラウドホスティング、データベース、ストレージ、生成 AI、文字起こし、監視、メール、CI/CD 等の外部委託先を利用する場合があります。これらの委託先には国外事業者または国外リージョンが含まれる可能性があり、個人データが日本国外で保存または処理される場合があります。

当社は、委託先の用途、扱うデータ、所在地、越境移転可能性を台帳化し、契約および運用上必要な安全管理措置を講じます。導入先から求められた場合、利用中の委託先一覧、データ所在地、保存期間、削除・復旧手順を提供します。

### 5.2 導入先へ確認すること

- 個人情報保護規程上、国外委託・越境移転の事前承認が必要か。
- 学校法人、自治体、上場企業の内部規程で国内リージョン必須条件があるか。
- 生成 AI / STT provider への入力制限があるか。
- 保護者・生徒への説明文に委託先または越境移転説明を含める必要があるか。
- 契約書または DPA で再委託先の事前承諾・事後通知のどちらが必要か。

## 6. データ種別別の国外処理可能性

| データ種別 | 国外処理可能性 | 管理策 |
| --- | --- | --- |
| 生徒・保護者情報 | あり得る | DB / hosting / backup の所在地を台帳管理 |
| 面談音声 | あり得る | 短期保存、STT 利用範囲限定、retention cleanup |
| 生文字起こし | あり得る | 保存期間短縮、生成・レビュー後 cleanup |
| structured artifact | あり得る | 正本としてアクセス制御、削除手順管理 |
| 保護者レポート | あり得る | 共有履歴と配信証跡を保持 |
| 監査ログ | あり得る | 本文・音声の過剰保存を避ける |
| 請求情報 | 原則低い | 直販・請求書払いで国内事務を基本とする |

保存期間と cleanup は [data-retention-policy.md](./data-retention-policy.md) を参照する。

## 7. 生成 AI / STT provider の扱い

生成 AI または STT provider を利用する場合は、次を導入先へ説明できる状態にする。

- 利用目的は文字起こし、要約、面談ログ生成、保護者向けレポート生成に限定する。
- 入力される可能性があるデータは、音声、文字起こし、面談文脈、固有名詞、レポート生成に必要な情報である。
- provider による学習利用、ログ保持、保存期間、データ所在地は provider の契約・設定に従う。
- 可能な場合、学習利用を無効化した API / business plan / enterprise plan を利用する。
- prompt には不要な個人データを含めない。
- 生成結果はレビュー対象であり、正本は structured artifact とレビュー済みレポートで管理する。

## 8. 委託先変更

### 8.1 変更に該当する例

- DB provider またはリージョンの変更
- Blob / object storage provider の変更
- 生成 AI / STT provider の変更
- 監視・ログ provider の追加
- メール・通知 provider の追加
- backup 保存先の変更

### 8.2 変更手順

1. 変更理由、対象データ、影響範囲を記録する。
2. 委託先の安全管理措置、所在地、越境移転可能性を確認する。
3. 契約・DPA・学校法人規程で必要な通知または承認を確認する。
4. 導入先へ通知する。
5. 移行後のデータ整合性、削除、バックアップ、監査ログを確認する。
6. 委託先台帳を更新する。

## 9. 審査回答テンプレート

### 9.1 データは日本国内に保存されるか

PARARIA は日本向け SaaS として国内顧客向けに運用されます。国内リージョンを選択できるサービスでは国内リージョンを優先します。一方、クラウド、生成 AI、監視、メール、CI/CD 等の一部委託先では国外事業者または国外リージョンが関与する可能性があります。利用中の委託先、用途、所在地、越境移転可能性は台帳で管理し、導入審査時に提示します。

### 9.2 委託先は誰か

委託先には、DB、object storage、hosting、生成 AI / STT、メール、監視、CI/CD、請求事務に関わる事業者が含まれます。実際に利用する委託先名とリージョンは導入環境ごとに台帳化し、契約または審査資料として提供します。

### 9.3 契約終了後にバックアップ内のデータはどうなるか

本番 tenant データは合意したエクスポート期間後に削除します。バックアップ内のデータは復旧目的で保持され、個別即時削除ではなくバックアップ保持期間の満了により削除されます。バックアップと復旧の詳細は [db-backup-recovery.md](./db-backup-recovery.md) に従います。

## 10. 運用証跡

次の証跡を保存する。

- 委託先台帳の更新履歴
- 委託先変更通知
- 導入先からの承認記録
- リージョン設定の確認結果
- AI / STT provider の学習利用設定確認
- backup 保存先と保持期間の確認
- 契約終了時の削除・バックアップ満了説明

## 11. 完了判定

この文書の運用は、次を満たした状態を完了とする。

- 導入先へ、国内向け SaaS であることと国外委託の可能性を矛盾なく説明できる。
- 委託先、用途、データ種別、所在地、越境移転可能性が台帳化されている。
- 決済ゲートウェイを利用しない前提が契約・請求運用と一致している。
- 保存期間、削除、バックアップ、復旧が既存文書と一致している。
- 委託先変更時の通知・承認・証跡手順がある。
