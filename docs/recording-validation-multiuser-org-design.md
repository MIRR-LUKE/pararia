# 録音バリデーション・マルチユーザー・組織設計（論点整理と設計メモ）

作成日: 2026-03-16  
目的: 短時間録音の扱い、検証ゲート、録音ロック、指導報告セッション、招待制組織について、**論点の妥当性・UX・技術の観点**で整理する。

---

## 0. 総評（このロジックどうか）

**結論: 方向性は非常に妥当です。**

| 論点 | 評価 |
|------|------|
| 録音完了 ≠ 生成開始（validation gate） | **推奨**。「誤タップ・誤停止」をそのまま課金・ジョブ化するのを防げる。 |
| ハード reject / ソフト警告の分離 | **良い**。一律タイムアウトだけにしない（VAD・無音・品質）は現場感覚と一致。 |
| セッションは残し invalid_recording | **必須に近い**。監査・改善・メモ退避に効く。 |
| ルーム全体ロックでなく録音だけ排他 | **UX・現場両面で最適解**。閲覧ロックはストレス・運用コストが高い。 |
| Heartbeat 付きロック | **標準的**。クラッシュ時の永久ロック回避に必須。 |
| 生成並列・同一 session 二重生成禁止 | **妥当**。コスト制御とユーザー混乱の両方に効く。 |
| **LLM 前ゲート**（ログとして無意味なら弾く） | **必須**。秒数は指標の一つに過ぎず、**コストの無駄**を止めるのはここ。 |
| 招待制・校舎・組織 | **B2B として正解**。公開サインアップは運用・セキュリティ面で不利になりやすい。 |

補足: 指導報告の **check-in / check-out と lesson_session_id** はデータモデルとロック設計の**中核**になります。ここを早めにスキーマに落とすと、後からの例外（orphan checkout）が扱いやすいです。

---

## 1. 短い録音と「保存」と「生成」の分離

### 1.1 状態機械（推奨）

提案されている流れを**そのまま採用**する価値が高いです。

```
recorded → uploaded → validating → accepted | rejected → processing（accepted のみ）
```

- **recorded**: クライアント上で録音終了。
- **uploaded**: ストレージ到達・整合性OK。
- **validating**: サーバで duration（例: **60 秒未満はここで reject**）/ VAD / 無音率 / 品質等を評価。
- **accepted / rejected**: ハード条件で機械判断。**rejected でも session row は残す**（下記）。
- **transcribing**: STT のみ（ゲート A 通過後）。
- **transcript_review**（概念）: STT 後に**テキスト薄さ**を判定。薄ければ **LLM には進めず終了**（§1.2a ゲート B）。
- **processing**: 構造化・要約など **LLM ジョブ**（ゲート B 通過後のみ）。

### 1.1a コスト最適化: 「秒数」ではなく「ログとして機能するか」で LLM 前に弾く

**目的**: 「短いから」ではなく **社内のログ品質の最低ラインを満たさない入力**を、**高コストな LLM（および可能なら高コスト STT）に渡す前**に止める。ユーザー文言では「短すぎます」になりがちだが、**内部の拒否理由は「ログとして成立しない」**（実質的な発話・内容がない／文字起こし不能）に寄せる。

**判定の主軸（推奨の考え方）**

| 観点 | ログとして無意味に近い例 | 備考 |
|------|--------------------------|------|
| **有効発話量** | VAD 上の実発話が極端に短い／ほぼ無音 | 「総秒数」より**発話に換算した量**が本質。 |
| **無音・ノイズ優位** | 無音率が高い／環境音だけ | 長時間録音でも**中身がない**なら弾く。 |
| **可聴性・デコード可否** | 極端に小さく正規化しても波形が実質フラット／デコード失敗・破損 | STT 以前に **unusable**。 |
| **テキスト化後の下限（任意）** | 安価 or 既存 STT で一度だけ短いプレビュー用転記し、**トークン数・語数が閾値未満** | コストを増やすなら **軽い STT / 短いチャンクのみ**に限定する設計が現実的。 |

**コスト段階（パイプラインの順序）**

1. **プレフライト（常に・安価）**: デコード、duration、チャネルエネルギー、無音比率、簡易 VAD、ファイル整合性。ここで **reject** できるものは **ここで止める**（LLM もフル STT も呼ばない）。
2. **必要に応じた中間（任意）**: 疑わしいケースだけ **短縮 STT・廉価モデル・長さ上限付き**で「最低限の文字があるか」だけ見る。ポリシーで「プレフライトだけで十分」とすることも可。
3. **本番 STT + LLM**: **accepted かつログとして成立する見込みが高いものだけ**。

**プロダクト文言との関係**

- UI では「短すぎる」と言いつつ、**内部 reason code** は `insufficient_speech` / `mostly_silence` / `unusable_audio` のように **秒数以外**も区別しておくと、サポートと閾値調整がしやすい。

### 1.1b 「内容の薄さ」はどう判定するか（正直な限界と実務ライン）

**結論**: 「面談として中身があったか」という**意味レベルの薄さ**を、**音声だけ・無料で完全に**判定するのは不可能に近い。実務では **段階を分けて**、コストと精度のバランスを取る。

#### 前提（限界）

- **音響だけ**では「うん」「はい」連発や、**長いが中身のないおしゃべり**を **人間と同じ粒度**では切れない。
- **意味の薄さ**を厳密にやるには、どこかで **言語に落とす（STT）か、極小の言語モデル**が必要になる。
- なので「薄い」とは設計上 **次のどれを指すか**を決める。
  - **A. ログに足りない（形式的）**: 文字が少なすぎる・発話がほぼない。
  - **B. 教育的に薄い（意味的）**: 事実・方針・宿題等の「荷物」がテキスト上ほぼ出てこない。

A は機械でかなりできる。B は**追加コスト**なしでは難しく、**軽い LLM 1 往復**か**ルール豊富なヒューリスティック**になる。

#### レベル 0：音響のみ（STT 前・安価）— 「形式的に中身がない」プロキシ

**できること**: 「ほぼ無音」「発話秒が極端に短い」「ノイズ床だけ」など。**LLM 不要**。

| 指標 | 薄いとみなす例 |
|------|----------------|
| **VAD 上の発話総量**（秒・フレーム数） | 閾値未満 |
| **音声区間の比率** | 全体に対して発話が極端に少ない |
| **RMS / ピーク** | 正規化後も実質無音に近い |
| **スペクトラル平坦さ・単調性**（任意） | 雑音床だけで変化がない |

→ ここで弾けるのは **「ログになる以前」** が大半。**内容の薄さ**というより **物理的な実体のなさ**。

#### レベル 1：廉価 STT のあと — 「テキストがログとして足りない」（おすすめの主戦場）

一度だけ **安い／短い** STT にかける（全文ではなく **先頭 N 秒・N 文字で打ち切り**でも可）。**高価なサマリー用 LLM にはまだいかない**。

テキストだけで見る **形式的な薄さ** の例（閾値はモード・言語で調整）:

| 指標 | 説明 |
|------|------|
| **正規化後の文字数・語数** | 日本語なら「有意味そうな文字数」下限（記号・空白除外）。 |
| **ユニーク文字・語の比率** | 極端に低い＝同じ語の繰り返しだけの可能性。 |
| **n-gram 反復率** | 同じフレーズの連続。 |
| **機能語だけ比率**（任意） | 「えー」「うん」「はい」止まりなら候補。**辞書＋簡易ルール**で十分なことが多い。 |
| **文区切り・句点数**（任意） | ゼロばかりでも、塾の面談では長文話し続けで句点なしもあるので**単独では弱い**。 |

→ **ここまでで「明らかにログとして薄い」**のかなりを止められる。**コストは STT 1 回分**（フル LLM より安い前提）。

#### レベル 2：境界線だけ「極小 LLM」— 意味的な薄さのざっくり判定（任意）

レベル 1 で **グレー**（短いが一応文章がある、など）に残ったもの**だけ**:

- **入力**: STT テキスト（全文または要約用に切ったチャンク）。
- **出力**: 構造化 JSON 例: `{ "substantive_enough": boolean, "reason_code": "…" }`
- **制約**: **max_tokens 極小**、**単一目的のプロンプト**（サマリー・構造化ログは書かせない）。

→ **フルパイプラインより遥に安いが 0 ではない**。ポリシーとして「レベル 1 通過＝そのまま本番 STT+LLM」にして、レベル 2 は後回しでもよい。

#### レベル 3：弾かずにソフト警告だけ

「形式的には通るが薄い疑い」→ **生成はする**＋ UI で「内容が薄くなる可能性」**（本文書 §1.3）**。

#### 実装順のおすすめ

1. **レベル 0 必須**（無音・発話量・品質）。
2. **レベル 1** で **軽量 STT + 文字数・反復・簡易パターン**まで一気にやると、**無駄な LLM** を一番止めやすい。
3. 運用で「まだ薄いのが通る」が残るなら **レベル 2 をグレー帯だけ**に追加。

#### reason code 例（内容薄の区分）

- `thin_content_acoustic` — 音響のみで不足。
- `thin_content_text` — 文字化け後の長さ・反復等で不足。
- `thin_content_semantic` — 極小 LLM が not substantive（導入時）。

### 1.2 ハード reject（自動で弾く）

#### 1.2a 運用ポリシー例（シンプル版・採用しやすい）

**面談・指導報告のどちらのモードも、次の 2 段ゲート**にすると説明がしやすく、コストとも折り合いがつきやすい。

**ゲート A — 長さ（音響・アップロード直後）**

| ルール | 挙動 |
|--------|------|
| **総録音時間が 1 分（60 秒）未満** | **エラー。生成パイプライン（STT 含む）には進めない。** |

- **意図**: 現場での「とりあえず押した」短尺を一括で止め、**文字起こしコストもかけない**。
- **補足**: 1 分以上でも、**無音優位・実発話がほぼない**（§1.1b レベル 0）は**別途 reject**（秒数だけに依存しない）。

**ゲート B — 文字起こし直後（薄さ）**

| ルール | 挙動 |
|--------|------|
| STT 完了後、**テキストが「ログとして明らかに薄い」**（§1.1b レベル 1 の閾値：文字数・反復・実質発話に相当する文字量など） | **ユーザーに「十分な内容が認められなかった」旨を伝え、それ以降（要約・構造化などの LLM）には進めない。** |

- **状態例**: `transcript_rejected_thin` / `thin_content_text` など。セッションは保持し、**再録音・アップロード**の導線を出す。
- **コスト**: ゲート A でかなり削れる。**ゲート B** は STT 1 回分はかかるが、**高価な下流 LLM はかけない**。

**ユーザー向け文言のイメージ（例）**

- ゲート A:  
  `録音が1分未満のため、面談ログ／指導報告の生成を開始できません。1分以上録音するか、音声ファイルをアップロードし直してください。`
- ゲート B:  
  `文字起こしの結果、会話として十分な内容が認められませんでした。生成処理はここで停止しています。もう一度録音するか、環境音・マイク音量を確認してください。`

（必要なら **「短いメモとしてテキストのみ保存」**は別フラグで許可する、は §1.4 の方針と併用。）

#### 1.2b モード別の微調整（任意）

1 分未満一律のほかに、**長さがクリアしたあと**は従来どおり **VAD・無音・品質**で差をつけてもよい（面談は長めが多い、指導報告はパート単位で短め、など）。**1 分未満は両モード共通エラー**で固定し、**1 分以上**だけモード別閾値にする形がわかりやすい。

| 段階 | 面談 | 指導報告 |
|------|------|----------|
| &lt; 60s | **共通エラー**（STT 未実行） | 同左 |
| ≥ 60s + 音響 NG | reject（無音・破損等） | 同左 |
| ≥ 60s、STT 後テキスト薄い | **LLM 以降は実行しない** + 明示メッセージ | 同左 |

**設計上の注意**

- **ゴールは「無駄な LLM」と「ユーザーが納得する説明」**。1 分未満は**運用ルールとして明文化**しやすい。
- 閾値（60 秒、テキストの薄さの数値）は **組織・校舎設定**にするとよい。
- クライアント単独では **VAD / 無音率を厳密に**しにくい。最終判断は**サーバ**（ffmpeg / 簡易エネルギー閾値等）を想定。

### 1.3 ソフト警告（弾かない）

- 生成は許可しつつ、**明確な一文**で期待値を下げる（提案のコピーで十分機能する）。
- UX のコツ: **モーダルで止めすぎず**、インラインまたはトースト + 「続ける」をデフォルトにすると離脱が減る。

### 1.4 Reject 時の挙動（UX ベストプラクティス）

「生成しない」だけにすると**ユーザーは何をすればよいか分からない**。以下を揃えるとよいです。

1. **理由コード**（UI 表示用の短い ID + 人間向けメッセージ）。
2. **次のアクション**（再録音が主。文言は提案の例のままでも良い）。
3. **セッションは `invalid_recording` 等で保持**（監査・サポート・改善分析）。
4. **オプション: 「短いメモとして保存（生成なし）」** — 現場価値が高い。データ上は `artifact_type=memo` と **LLM パイプラインを分ける**と安全。

### 1.5 アップロード時のルール

- **mime / 拡張子 / ファイル整合性 / duration（メタまたはデコード）**はサーバで必須。
- **上限時間**（面談 60 分・指導報告 10 分）は**ストレージ・コスト・ユーザー誤操作防止**の両方に有効。
- エラーメッセージは**行動可能**にする（分割・別ファイル・別パート等）。

## 2. 複数人同時利用とロック（UX + システム）

### 2.1 なぜ「録音だけ排他」が最適か（リサーチとの関係）

- **協調編集**（Google ドキュメント型）では楽観的ロックが多いが、**録音は「同時に 1 物理マイク・1 会話コンテキスト」**に近く、**悲観的ロック（事前ブロック）**の方がユーザー心理と一致しやすい。
- **ルーム全体をロック**すると、説明・確認・保護者レポートの下読みまで止まり、**認知負荷と不満が跳ねる**（提案の「閉じない」判断は UX 的に妥当）。

### 2.2 推奨ロックのスコープ

| ロック | 範囲 | 目的 |
|--------|------|------|
| **recording lock** | 1 生徒 × 同時 1 人のみ録音 | 衝突・二重アップロード・混乱の防止 |
| **report draft edit lock**（任意） | 同一ドラフトの編集中 | 上書き事故の防止。最初は「最後に保存した wins」でも可。 |

### 2.3 録音中の UI（提案の「誰が録音中か」）

**ベストプラクティス:**

- **録音者だけ**が「停止・取消」可能。他者は **録音ボタン disabled**。
- **名前 + アバター**（講師アイコン）で**誰がロック保持者か**を明示。  
  - 生徒一覧の生徒名横に小さいアバター＋「録音中」バッジは**一覧から状況が一目で分かる**ため有効（提案どおり）。
- **バナー**（生徒詳細トップ）:  
  `◯◯ が面談モードで録音中です。閲覧はできますが、新しい録音は開始できません。`
- **管理者: 強制解除**は監査ログ付き（誰が・いつ・理由）。

### 2.4 `student_recording_lock`（技術仕様の補足）

提案のフィールドは十分。追加検討:

- **mode**: `INTERVIEW` | `LESSON_REPORT`（指導報告ならチェックイン/アウトのどちらかも subtype で持てる）
- **lock_token**: クライアントが release 時に持参（自分のロックだけ解放）
- **expires_at / heartbeat**:  
  - 10s heartbeat / 30s 失効は**初期値として妥当**（校舎の回線が不安定なら 45–60s に緩めることも）。

### 2.5 処理中（processing）の扱い

**提案どおり推奨:**

- **生成中でもルーム閲覧は可能**。
- **録音**は **active recording lock がないときのみ**開始可能（並行ジョブは許容）。
- UI で **「処理中」の進捗**を別表示（キュー位置は出せると安心感が増す）。

### 2.6 指導報告の `lesson_session_id` とセーフティ

- **チェックイン開始時に lesson session 確定** → チェックアウトは同一 ID に紐づく、は一貫性が高い。
- **check-in だけ**: `incomplete`（明示）。
- **checkout だけ**: `orphan_checkout` + **手動紐付け or 室長承認**の導線があると運用が回る。
- **同一チェックイン/アウトは原則同一講師** / **takeover は管理者**は現場と整合。

### 2.7 生成ジョブの並列と一意性

- 別生徒・別 completed session の並列生成は**積極的に許可**。
- **同一 `(session_id, artifact_type, generation_attempt または idempotency key)`** の二重投入を**DB 一意制約 or ジョブキュー**で防ぐ。

## 3. ログイン・校舎・招待制

**結論: 塾向けでは招待制・組織単位が標準に近い。**

- 公開サインアップは **スパム・誤登録・データ隔離の事故**が増えやすい。
- 推奨フロー（概念）:
  1. **Organization（事業者 / 塾法人）**
  2. **Campus（校舎）** — 権限・データスコープの単位
  3. **招待メール / 招待リンク**（期限・1 回限り・ロール指定）
  4. **ロール**: 室長 / 講師 / 管理者（強制ロック解除・監査）

---

## 4. UX チェックリスト（実装・レビュー用）

- [ ] reject 時は**理由 + 次の一手**（再録音 / メモ保存）が常に見える。
- [ ] soft warning は**邪魔にならない位置**（インライン推奨）。
- [ ] 他人が録音中でも**閲覧・既存ログ・（方針次第）レポート生成**が可能。
- [ ] 録音ボタン disabled 時は**なぜ disabled か**を必ず表示（名前・モード）。
- [ ] ロックは**自動失効**し、サポート負荷を上げない。
- [ ] 管理者操作は**監査ログ**。
- [ ] 生徒一覧で**録音中インジケータ**（アイコン・ツールチップ）。

## 5. 未決論点（プロダクトで決めるべきこと）

1. **soft warning** をユーザー設定で「毎回確認」「一度スキップ」にできるか。
2. **メモ保存**を Tier 別に制限するか（ストレージポリシー）。
3. **processing 中の二件目録音**を校舎ポリシーで禁止するか（稀だが混乱する現場もある）。
4. **VAD・品質閾値**の初期値を全国一律か校舎別か。
5. **指導報告の orphan checkout**の SLA（何日以内に紐付け必須か）。

### 6.1 実装状況（サーバ・ゲート A/B）

| 項目 | 状態 |
|------|------|
| ゲート A（最小事前録音秒数・STT 前） | ✅ `lib/recording/validation.ts` + `POST .../parts` |
| ゲート B（文字起こし／手入力テキストの薄さ・LLM 前） | ✅ 同一 |

| 録音ロック・heartbeat | ✅ `StudentRecordingLock`・`lib/recording/lockService.ts`・`GET/POST/PATCH/DELETE /api/students/[id]/recording-lock`・音声 `POST .../parts` は `lockToken` 必須（手入力テキストのみは不要） |

## 7. 一文サマリー

**録音は「検証してから生成」、マルチユーザーは「閲覧は開く・録音だけロック」、組織は「招待制」。この 3 点セットは PARARIA の利用シーン（塾・複数講師）と整合し、UX と運用コストのバランスが良い。**
