# 保護者レポート生成で、壊れた面談ログを markdown fallback で通さない

## 何がつらいか

- いまは `artifactJson` が壊れていても、`summaryMarkdown` から擬似的に材料を作って保護者レポートを続けられる
- その結果、上流の面談ログ生成が壊れているのに、保護者レポートだけ何となく出てしまう
- 問題の本体が隠れて、調査が長引く

## 目標

- 保護者レポートは「正常な面談ログ artifact があること」を前提にする
- 壊れたログや旧形式の fallback では静かに進めず、分かりやすく止める
- 上流が壊れていたら、その場で気づけるようにする

## やること

- 保護者レポート生成前に、選択ログごとに `artifactJson` の妥当性を確認する
- 妥当でないログが 1 本でもあれば、どのログが原因か分かる形で生成を止める
- `summaryMarkdown` だけから保護者レポート材料を再構成する経路をやめる
- 回帰テストを追加して、artifact 不正時にエラーで止まることを確認する

## 完了条件

- 保護者レポート生成は正常な artifact 付きログだけ通る
- 壊れたログでは明確なエラーになる
- 専用の回帰テストが追加されている
