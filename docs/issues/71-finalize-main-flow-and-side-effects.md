# finalize 成功後に、後続処理の失敗で面談ログの状態を壊さない

## 何がつらいか

- 面談ログ本文の保存が終わったあとも、次回面談メモ enqueue や session 同期などを同じ流れで続けている
- その後で落ちると、本文は保存済みなのに `PROCESSING` や `ERROR` に戻るねじれが起きうる
- 「保存されたかどうか」がぶれると、調査が一気に難しくなる

## 目標

- `FINALIZE` の主目的を「面談ログ本文の確定保存」に絞る
- その保存が終わったら、以後の補助処理が失敗しても会話ログの主状態は壊さない
- 次回面談メモや worker stop は best-effort にして、失敗しても主成果物は守る

## やること

- 面談ログ本文の保存、job 完了、会話 status 確定までを主経路としてまとめる
- `next meeting memo` の enqueue、session 反映、worker stop は主経路から切り離す
- 補助処理が失敗した時は警告ログだけ残し、会話ログ本文の確定状態は保つ
- 回帰テストを追加して、主成果物保存後の side effect 失敗で状態が戻らないことを確認する

## 完了条件

- 面談ログ本文が保存された後は、補助処理の失敗で `PROCESSING` / `ERROR` に戻らない
- 警告ログで追跡できる
- 専用の回帰テストが追加されている
