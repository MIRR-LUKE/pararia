# Student Room UX/UI 超改善企画（全体版）

作成日: 2026-03-15  
更新: 2026-03-15（リサーチ反映・全体UX拡張）  
対象: アプリ全体（`/app` 配下）— ダッシュボード・生徒一覧・生徒詳細・ログ・レポート・設定

---

## 1. 背景と課題

現状の Student Room は情報量が多く、**「録音」「会話ログ」「指導報告書」** というコア体験がページの下の方や別カードに埋もれている。加えて、ナビゲーション・視覚階層・空状態・ローディング・一貫性・アクセシビリティなど、**全体のUX/UI** をまとめて見直す必要がある。

### 現状の主な課題

| 領域 | 課題 |
|------|------|
| **録音** | 面談・指導報告の録音がファーストビューにない。スクロールしないとたどりつけない |
| **会話ログ** | コミュニケーション履歴が下にあり、会話ログ一覧／直近ログへの明確な入口がない |
| **指導報告書** | 指導報告セッション完了後、「このセッションで報告書を作る」導線が画面にない |
| **ナビ** | サイドバーに「会話ログ」がない。生徒詳細→ログへの文脈が弱い |
| **情報量** | 生徒詳細が縦長で、何をすべきか（Next Best Action）が埋もれる |
| **空・ローディング** | 「読み込み中...」のみ。空状態にCTAやサンプル提示が弱い |
| **一貫性** | ページごとにレイアウト・余白・カードの使い方がばらつく |
| **レスポンシブ** | サイドバー固定幅・コンテンツ幅のモバイル対応が未整備 |

## 2. 優先方針（3本柱 ＋ 全体UX）

### 2.1 コア3本柱（最優先）

| 優先度 | 方針 | 狙い |
|--------|------|------|
| **1** | **圧倒的にすぐ録音ボタンがある** | 開いた瞬間に「録音する」が選べる。スクロール不要で録音開始できる |
| **2** | **会話ログにたどりつける** | 生徒詳細から会話ログ一覧／直近ログへ 1〜2 クリックで到達する |
| **3** | **指導報告セッションから指導報告書をつくれる** | 指導報告を記録したあと、「このセッションで報告書を作る」が明示され、ワンアクションでレポート作成に進める |

### 2.2 全体UXの柱（リサーチ反映）

- **意思決定駆動**: ダッシュボードを「見るだけ」ではなく「今日やることを決める」画面にする。North Star に近い指標と Next Best Action を左上〜上部に配置する。
- **プログレッシブディスクロージャー**: 初回表示は「録音・ログ・報告書」に集中し、詳細（話題カード・次の一手・プロフィール等）は折りたたみ or タブで後から開く。
- **明確な視覚階層**: 重要度の高い要素をサイズ・色・余白で差をつける（Squint Test で判別できるレベル）。
- **空状態・ローディング**: 空は「次にやること」を教える。ローディングはスケルトン or 進捗付きで不安を減らす。
- **一貫したデザインシステム**: 余白（8px ベース）、カード、ボタン、バッジを共通化し、ページをまたいで同じルールで並べる。
- **アクセシビリティ**: キーボードフォーカス可視（:focus-visible）、ARIA、スクリーンリーダー対応を前提にする。

## 3. コア3本柱の具体的な改善（前回案の要約）

### 3.1 録音を最前面に出す

- **アクションヒーロー**: 生徒詳細の `AppHeader` 直下に 1 ブロック追加。左に「面談を録音」、右に「指導報告を記録」。クリックで既存の StudentRecorder / LessonReportComposer にスクロール＋フォーカス。
- 既存の録音カードは 2 番目以降に配置し、「詳細はこちら」で展開する形でも可。

### 3.2 会話ログにたどりつける

- 生徒詳細の上部（身份カード横 or ヒーロー右端）に **「会話ログを見る」** を配置。直近 1 件のログ詳細 `/app/logs/[id]` へ遷移（`room.sessions[0]?.conversation?.id` を利用）。または直近ログをカードで 1 件表示し、クリックで詳細へ。

### 3.3 指導報告セッション → 指導報告書

- LessonReportComposer の完了時（`conversationId` 確定時）に **「このセッションで保護者レポートを作成」** を「ログ詳細へ」の横に追加。
- クリックで `/app/reports/[studentId]?sessionIds=xxx` に遷移し、到着時にそのセッションを選択済みにする。レポートページで `sessionIds` クエリを読み取り初期選択に反映。

## 4. ナビゲーション・情報アーキテクチャ

### 4.1 サイドバー

- **現状**: ダッシュボード / 生徒一覧 / レポート / 設定のみ。「会話ログ」がない。
- **改善案**:
  - **会話ログ** を追加: `/app/logs` を「会話ログ一覧」として用意し、サイドバーから遷移できるようにする。一覧では生徒フィルタ・日付ソートを用意。
  - または「生徒一覧」のサブとして「会話ログ」を出さず、**生徒詳細内の「会話ログを見る」で十分** とする場合は、サイドバーは現状のままでも可（導線を生徒詳細に集約する方針）。

### 4.2 パンくず・コンテキスト

- 生徒詳細・ログ詳細・レポート詳細で **パンくず** を表示: `生徒一覧 > 山田太郎 > 会話ログ` のように、今どこにいるかを示す。
- AppHeader の `subtitle` を「コンテキスト＋次の一手」に寄せる（例: 「面談を録音するか、直近ログを確認」）。

### 4.3 生徒詳細のタブ化（オプション）

- 生徒詳細を **「概要」「会話ログ」「レポート」** のタブに分けると、会話ログへの到達がさらに明確になる。タブ「会話ログ」でその生徒のセッション一覧を出し、各行からログ詳細へ。

## 5. 視覚階層・意思決定駆動デザイン

### 5.1 North Star と F パターン

- **左上〜上部**: そのページで「今日の一番大事な数字 or アクション」を置く。
  - 生徒詳細: 録音 CTA ＋ 直近ログへのリンク。
  - ダッシュボード: 「フォロー優先」を左カラムの上にし、KPI は 1 行にまとめる。
- **F パターン**: 重要な要素を左から右、上から下の視線の流れに乗せる。長いリストは左にラベル・右にアクション。

### 5.2 Squint Test

- 画面をぼかして見たときに、「録音」「会話ログ」「報告書」に相当するブロックが他より目立つようにする。色・サイズ・余白で差をつける。

### 5.3 カード・セクションの優先度

- **高**: アクションヒーロー、直近ログ、保護者レポート生成。
- **中**: 身份カード、コミュニケーション履歴、指導報告／面談の詳細 UI。
- **低**: おすすめの話題、次の一手、プロフィール、要確認固有名詞 → 折りたたみ or 2 列目以降に配置。

## 6. プログレッシブディスクロージャー

### 6.1 生徒詳細の折りたたみ

- 「おすすめの話題」「質問」「次の一手」「今週の変化」を **アコーディオン** にし、初期は 1 つだけ開く or すべて閉じる。「開く」で詳細を表示。
- プロフィール・要確認固有名詞も「詳細を見る」で開く形にすると、ファーストビューが録音・ログ・報告書に集中する。

### 6.2 ダッシュボード

- KPI と「フォロー優先」を前面に。その他のリストは「もっと見る」で展開するか、2 列目に回す。

### 6.3 録音 UI

- ヒーローは「録音を開始」のみ。ファイルアップロード・テキスト入力は「詳細」カード内に残す（既存の StudentRecorder / LessonReportComposer）。

## 7. ローディング・空状態・エラー

### 7.1 ローディング

- **現状**: 「読み込み中...」のテキストのみ。
- **改善**:
  - 一覧系（生徒一覧、コミュニケーション履歴）: **スケルトン**（カード形のプレースホルダー）を表示。 perceived performance が上がる。
  - 録音の「処理中」: **進捗バー or 段階表示**（文字起こし中 → AI 生成中）を既存のまま明確に。 indeterminate なスピナーのみにしない。
  - 反応閾値: 0.2 秒で何か反応、1 秒以内でフローが続くように。10 秒以上かかる処理は「バックグラウンドで続行」＋完了通知を検討。

### 7.2 空状態

- **現状**: 「まだセッションがありません」などテキストのみで終わりがち。
- **改善**:
  - 空状態には **理由の説明＋CTA** を必ず入れる。例: 「まだ面談記録がありません。上の「面談を録音」から追加すると、ここにログが並びます。」
  - 可能なら **サンプル or プレビュー**（「記録があるとこんな風に表示されます」）を軽く見せる。初週の離脱を防ぐ。
  - 生徒が 0 件: 「最初の生徒を追加」ボタンを大きく。登録フローへ 1 クリックで進める。

### 7.3 エラー

- API 失敗時: `alert` だけでなく、**画面上のバナー or インライン** でメッセージを表示。リトライボタンを用意する。
- 録音・権限エラー: 「HTTPS または localhost が必要です」「マイクを許可してください」など、**次のアクション** を一文で示す。

## 8. 一貫性・デザインシステム

### 8.1 スペーシング・グリッド

- **8px ベース** で統一: 4, 8, 12, 16, 24, 32, 40, 48。`gap`・`padding`・`margin` をこのスケールに寄せる。
- **コンテンツ幅**: `max-width: 1200px` は現状維持。カード内は 12 カラムグリッドを意識し、2 列レイアウトを崩さない。

### 8.2 コンポーネント

- **Button**: primary / secondary / ghost の用途を統一。録音は「危険色」で目立たせる。
- **Card**: title / subtitle の有無、padding を全ページで同じにする。
- **Badge**: tone（neutral / low / medium / high）をステータスと対応させ、色を共通化。
- **AppHeader**: 各ページで `title` / `subtitle` / `actions` の役割を揃える。`actions` にはそのページの主アクションを置く。

### 8.3 タイポグラフィ・色

- 見出しレベル（h1/h2/h3）をページ間で統一。重要度の高い数字はフォントサイズ・ウェイトで強調。
- 主アクション（録音）: 赤系。セカンダリ（指導報告・ログ）: 青系。成功・完了: 緑。エラー: 赤。を CSS 変数で定義し、流用する。

## 9. アクセシビリティ

### 9.1 キーボード・フォーカス

- **WCAG 2.4.7 Focus Visible**: すべてのインタラクティブ要素に **:focus-visible** で 2〜3px のアウトライン or ボーダーを付ける。contrast 3:1 以上。
- モーダル・ドロワーは開いたらフォーカスをトラップし、閉じたら元のトリガーに戻す。

### 9.2 セマンティクス・ARIA

- 録音ボタン: `aria-label="面談を録音"`、`aria-pressed` で録音中かどうか示す。
- セクションには `aria-labelledby` で見出しと紐づける。アコーディオンには `aria-expanded`。
- ローディング中は `aria-live="polite"` で「処理中」を読み上げさせる。

### 9.3 コントラスト・テキスト

- 本文・ラベルは背景とのコントラスト比 4.5:1 以上。重要なラベルは 7:1 を目標に。

## 10. レスポンシブ・モバイル

### 10.1 ブレークポイント

- 768px 以下: サイドバーをハンバーガー or 下タブにし、メインコンテンツを全幅に。
- 生徒詳細の 2 列（例: 話題＋質問）は 1 列にスタック。メトリクスは 2x2 や 1 列に。

### 10.2 タッチ

- 録音ボタン・主要 CTA は **最小 44x44px** のタッチ領域に。余白で囲んでクリックしやすくする。

### 10.3 録音のモバイル

- マイク権限のプロンプトが分かりやすいように、事前に「このアプリは録音にマイクを使います」と一文表示する案を検討。

## 14. まとめ

- **コア3本柱**: 録音を最前面に、会話ログに 1〜2 クリックでたどりつけ、指導報告セッションから保護者レポートを 1 アクションで作れるようにする。
- **全体UX**: 意思決定駆動・プログレッシブディスクロージャー・視覚階層・空状態・ローディング・一貫性・アクセシビリティ・レスポンシブを、リサーチに基づいて企画に組み込んだ。
- **実装**: Phase 1 で 3 本柱を実装し、Phase 2〜4 でナビ・階層・空状態・一貫性・a11y・モバイルを順に整える。

この方針で、Student Room 以降の「録音 → ログ確認 → 報告書作成」が明確になり、かつ全体のUX/UI が「使える・分かる・迷わない」レベルまで引き上げられる。
