# PARARIA 運営管理コンソール仕様計画

作成日: 2026-04-26

## 目的

`/admin` はクライアント校舎向けの設定画面ではなく、PARARIA 運営側が数十から数百校舎を横断して、安全に監視・支援・復旧・監査するためのバックオフィスにする。

最初の画面で答えるべき問いは次の 3 つ。

- いま要対応の校舎・ジョブ・障害はどれか
- どの校舎で、誰に、何が起きているか
- 運営者が何をしてよく、何をするには承認や理由が必要か

## 調査から採用する原則

- Atlassian Administration は、組織横断の監視、ユーザー管理、組織ライフサイクル管理を「一箇所」に集めている。PARARIA でも校舎単位の設定ではなく、運営者用の中央管理にする。
- Google Cloud IAM は、組織構造に合わせたリソース階層、ラベル、監査を前提にしている。PARARIA も `契約顧客 -> 校舎 -> ユーザー/生徒/面談/ジョブ` の階層を明示する。
- Microsoft Entra PIM は、高権限操作に MFA、理由、承認、期限付き権限を組み合わせる。PARARIA もジョブ再実行、ユーザー停止、データ復元などの操作は理由・監査・必要に応じた二重承認を必須にする。
- Stripe Organizations は、組織レベルとアカウントレベルのロールを分け、チーム一覧の CSV 出力、2FA、セキュリティ履歴を持つ。PARARIA も運営者ロールと校舎内ロールを分離する。
- Google SRE の監視設計では、個別ノイズではなく高レベルのサービス目標から見て、必要なときに詳細へ掘る。PARARIA も「失敗一覧の羅列」ではなく、SLO、遅延、詰まり、影響校舎数から見る。
- GOV.UK の admin interface 指針では、DB 行をただ表示せず、実際の管理タスクに合わせる。PARARIA も非エンジニア情シスが使う前提で、作業別の画面にする。
- Nielsen Norman Group の usability heuristics では、状態の可視化、現実の言葉との一致、エラー予防、思い出させるより認識させること、最小限の情報設計が重視されている。PARARIA も専門用語や内部コードを前面に出さず、状態・原因・次の一手を日本語で見せる。
- GOV.UK の UI writing 指針では、利用者は細かい説明を読まずに走査する前提で、短く直接的な文言を推奨している。PARARIA も説明文で補うのではなく、画面構造とラベルだけで迷わない設計にする。
- GOV.UK の table 指針では、大きな表は読みにくく、重要情報を先頭列・上部に置くべきとされている。PARARIA も「全項目一覧」ではなく、初期表示は対応判断に必要な列だけに絞る。
- W3C WCAG 2.2 を最低基準に、キーボード操作、フォーカス表示、コントラスト、クリック領域を守る。色だけで状態を伝えない。

## UX/UI 設計方針

この管理画面は、エンジニアのための監視ツールではなく、PARARIA 運営、CS、非エンジニア情シスが毎日使える業務画面にする。

目標:

- 初見でも「どこを見るか」「次に何を押すか」がわかる
- 研修なしでも、校舎検索、状態確認、一次対応、エスカレーションができる
- 危険操作は簡単に見つかるが、誤操作では実行できない
- PII や面談本文を見なくても、ほとんどの一次対応が完了する
- 画面を見れば、対応済み・未対応・保留・失敗の違いがわかる

基本ルール:

- 1 画面 1 目的にする。ホームは「要対応の把握」、校舎一覧は「探す」、校舎詳細は「状況を見る」、ジョブ画面は「詰まりを直す」。
- 初期表示は最大 3 領域までにする。ヘッダー、主要アクション、一覧または詳細だけを基本形にする。
- ダッシュボードの数値カードは最大 4 つまでにする。大量のカードやグラフで埋めない。
- グラフは原則 1 画面 1 つまで。日次推移を見る必要がある場合だけ使う。
- 表の初期列は 6 列前後に抑える。詳細は行クリック後の詳細パネルに逃がす。
- すべての状態ラベルは色だけでなく文字を持つ。例: `要対応`, `処理中`, `正常`, `保留`, `停止中`。
- エラー表示は内部コードを主語にしない。例: `STT_WORKER_TIMEOUT` ではなく `音声文字起こしが時間内に終わっていません`。
- ボタンは画面ごとに主操作を 1 つだけ強調する。危険操作は詳細画面か確認ダイアログ内に置く。
- disabled ボタンだけを置かない。押せない理由と解決方法を横に出す。
- パンくず、校舎名、現在の対象を常に表示し、今どの校舎を見ているか迷わせない。
- ローディング、空、エラー、権限不足は専用状態を持つ。空画面でも次の行動を 1 つだけ示す。
- テキストは短くする。説明文で UI を補う状態になったら、画面構造を直す。

視覚設計:

- ベースは白/薄いグレー/濃い文字の業務画面にする。装飾的なグラデーション、巨大なヒーロー、派手なカード群は使わない。
- 色は状態補助に限定する。赤は緊急、黄は注意、緑は正常、青は情報に固定する。
- 余白は広げすぎない。SaaS の管理画面として、一覧性と読みやすさを優先する。
- 行高、ボタン高、入力高を揃える。クリック領域は小さくしない。
- アイコンは補助として使い、必ずラベルまたはツールチップで意味を補う。
- 重要な数字は大きくしすぎない。数字より「要対応かどうか」を目立たせる。

操作設計:

- 基本導線は `検索 -> 校舎詳細 -> 状態確認 -> 必要なら操作` に統一する。
- 危険操作は `操作内容 -> 影響範囲 -> 戻せるか -> 理由入力 -> 実行` の順に確認する。
- 一括操作は初期フェーズでは作らない。必要になったら件数プレビューと承認つきで追加する。
- 「代理閲覧」や「本文表示」は通常操作とは別扱いにし、理由と監査を必須にする。
- エンジニア向け情報は折りたたむ。初期表示では原因分類、対応状況、次の手だけを出す。

文言ルール:

- 画面名は業務の言葉にする。例: `校舎`, `要対応`, `ジョブ`, `監査`, `契約`, `端末`。
- 内部語を避ける。`tenant` は `校舎`、`organization` は文脈に応じて `顧客` または `校舎`、`worker` は `処理基盤`。
- エラーメッセージは `何が起きたか`、`影響`、`次にすること` の順で書く。
- 確認ダイアログは短くする。長い説明ではなく、対象・影響・理由欄・実行ボタンに分ける。
- ボタン文言は動詞にする。例: `再実行する`, `停止する`, `理由を記録して表示する`。

アクセシビリティ:

- キーボードだけで検索、フィルタ、詳細確認、操作確認まで進める。
- フォーカスリングを必ず表示する。
- テキストコントラストは WCAG 2.2 AA を満たす。
- 状態を色だけで伝えない。状態バッジには必ず文字を入れる。
- 表は 1 ヘッダー行、1 セル 1 情報を基本にする。
- スクリーンリーダー向けに、行アクションは `田中校の詳細を見る` のように対象を含める。

ユーザビリティ確認:

- 実装ごとに、少なくとも次の 5 タスクを人が見て確認する。
- `校舎名で検索して詳細を開ける`
- `要対応の校舎を見つけ、原因分類を説明できる`
- `詰まったジョブを見つけ、再実行前の影響範囲を確認できる`
- `端末を revoke する前に対象校舎と端末名を確認できる`
- `監査ログから、誰がいつ何をしたか説明できる`

実装後レビューでは [admin-console-review-checklist.md](./admin-console-review-checklist.md) を使う。静的な補助確認は `npx tsx scripts/check-admin-console-simplification.ts` で実行できるが、非エンジニア向け 5 タスク確認を置き換えるものではない。

## 現状とのズレ

現在の実装は `User.organizationId` を前提にしており、ログインユーザーの所属組織の `settings` 由来スナップショットを `/admin` に表示している。

これは「校舎内の管理者が自校舎を見る」構造であり、PARARIA 運営者が全校舎を横断して見る構造ではない。

不足しているもの:

- 運営者専用ロールと校舎内ロールの分離
- 顧客法人、校舎、契約、運用状態の階層
- 全校舎横断の検索、フィルタ、健全性一覧
- 運営操作の理由、承認、監査ログ
- 校舎詳細からジョブ、ユーザー、端末、面談、監査へ掘る導線
- PII と面談本文を初期表示しない情報設計
- 失敗ジョブ再実行など危険操作の安全装置

## 情報アーキテクチャ

推奨する管理単位:

- CustomerAccount: 契約主体。法人、請求先、担当者、契約状態を持つ。
- Campus: 実運用単位。校舎、教室、ブランド校など。既存 `Organization` は短期的には Campus 相当として扱う。
- Organization: 現行互換のテナント境界。将来的には `customerAccountId` と `campusCode` を追加するか、`Campus` を新設して移行する。
- PlatformOperator: PARARIA 運営者。校舎内ユーザーとは別管理。
- PlatformAuditLog: 運営者操作の監査。理由、対象、変更前後、IP、User-Agent、承認情報を保存する。

## `/admin` の主要画面

1. 司令塔ダッシュボード

- 初期表示は `要対応`, `処理中`, `正常`, `停止中` の 4 状態だけを上部に出す。
- メイン領域は `今日の要対応` 一覧にする。グラフ中心の画面にはしない。
- 一覧は `状態`, `校舎`, `何が起きているか`, `経過時間`, `担当`, `次にすること` に絞る。
- SLO や p95 は折りたたみの `サービス状態` に置き、非エンジニアの初期判断を邪魔しない。
- 最近の高権限操作は下部に 5 件だけ出す。詳細は監査画面へ遷移する。

2. 校舎ディレクトリ

- 初期フォーカスは検索欄に置く。校舎名、顧客名、管理者メール、契約番号、校舎コードで探せる。
- フィルタは初期 5 つだけにする。`すべて`, `要対応`, `稼働中`, `導入準備中`, `停止中`。
- 詳細フィルタは `さらに絞り込む` の中に置く。契約更新、利用急増、失敗多発などは初期表示に出さない。
- 一覧列は `状態`, `校舎`, `顧客`, `生徒数`, `最終利用`, `担当`, `詳細` に絞る。
- CSV 出力は監査つき。PII 過多の列は初期非表示。

3. 校舎詳細

- 上部に `校舎名`, `状態`, `担当CS`, `最終利用`, `未対応件数` を固定表示する。
- 最初のタブは `概要`。契約、担当者、利用量、直近アラートを 1 画面で読める量にする。
- タブは `概要`, `ユーザー`, `ジョブ`, `端末`, `監査`, `メモ` に限定する。
- 生徒/面談は件数と状態のみを初期表示する。本文や音声は `理由を記録して表示` の操作を通す。
- 危険操作はページ上に常時露出しない。対象セクションの詳細パネルから実行する。

4. オペレーションキュー

- 初期表示は `要対応のみ` にする。正常終了済みジョブは検索時だけ出す。
- 横断ジョブ一覧は `状態`, `校舎`, `処理`, `経過時間`, `原因`, `次にすること` に絞る。
- 操作は `詳細を見る` を先に置く。再実行、キャンセル、保留解除、Runpod 状態確認は詳細内に置く。
- 安全装置: 単一ジョブ操作を基本にし、複数操作はプレビュー、理由入力、影響件数確認、監査を必須にする。

5. 監査・ログ検索

- 初期表示は検索フォームと最近の重要操作 20 件にする。
- 検索軸は `期間`, `操作者`, `校舎`, `操作種別`, `結果` に絞る。IP や targetId は詳細検索に置く。
- 重要操作は詳細画面で変更前後、理由、承認者、関連チケットを表示
- CSV/JSON エクスポートは権限限定、全件監査

6. データライフサイクル

- 初期表示は `削除依頼`, `復元依頼`, `期限切れ間近`, `バックアップ確認` の 4 キューに分ける。
- 各キューは件数と最古の未処理日だけを上部表示し、詳細は一覧へ掘る。
- 上場企業の監査で説明できるよう、手順書と実行証跡へリンクする。

7. 契約・運営メタデータ

- 請求は外部の請求書払い前提。アプリ内課金は不要。
- 必要なのは契約状態、契約更新日、請求先、営業担当、CS担当、利用上限、特記事項。
- 契約画面は財務システムの代替にしない。校舎運用に必要な参照情報だけに限定する。

## 運営者ロール

- PLATFORM_OWNER: 全権限。運営者管理、危険操作、承認。
- OPS_ADMIN: ジョブ・障害・復旧操作。顧客契約情報は閲覧のみ。
- SUPPORT_LEAD: 校舎支援、ユーザー/端末支援、サポートメモ。
- CUSTOMER_SUCCESS: 利用状況、契約メタデータ、校舎状態。危険操作不可。
- READONLY_AUDITOR: 監査閲覧、エクスポート限定。
- ENGINEER_ONCALL: 障害時の技術ログ、Runpod/STT、ジョブ復旧。個人情報本文は原則マスク。

## セキュリティ要件

- 本番は `admin.pararia...` のような管理者サブドメインを正式入口にする。
- 通常アプリの `/app/settings` はクライアント校舎向け設定として残す。
- `/admin` は `PlatformOperator` のみ。校舎内 `ADMIN` だけでは入れない。
- 高権限操作は理由入力を必須化する。
- 影響が大きい操作は二重承認または短時間の権限昇格にする。
- 初期表示では音声、面談本文、保護者連絡先などの PII を出さない。
- 全 admin API は対象校舎を明示し、暗黙の `session.user.organizationId` に依存しない。
- admin の write 操作は `PlatformAuditLog` に必ず記録する。

## `/app/settings` との責務境界

`/app/settings` は導入先校舎が自校舎の設定を管理する場所として残す。扱ってよいものは、校舎内ユーザー招待、校舎設定、Teacher App 端末の校舎内管理、通常の権限範囲で完結する設定に限る。

`/app/settings` に置かないもの:

- 全校舎横断の検索、監視、集計
- 本番ジョブの再実行、キャンセル、保留解除
- Runpod / STT / LLM 基盤の横断操作
- 代理閲覧、面談本文表示、音声表示、保護者連絡先の横断確認
- 監査ログ export、契約メタデータ、CS サポートメモ
- platform operator の管理

これらは `/admin` の platform admin shell に集約し、必要に応じて理由入力、影響確認、監査を通す。

## 本番入口と環境変数

- `PARARIA_ADMIN_HOSTS`: admin として扱う host の allowlist。例: `admin.pararia.example`。
- `PARARIA_ADMIN_BASE_URL`: 通常 host の `/admin` から誘導する正式 admin URL。例: `https://admin.pararia.example`。
- `PARARIA_ADMIN_OPERATOR_EMAILS`: 移行・緊急用の operator email allowlist。最終的な正は `PlatformOperator` / `PlatformRole` に置く。

本番では admin host の `/` を `/admin` に寄せ、通常アプリ host の `/admin` は `PARARIA_ADMIN_BASE_URL` へ redirect できるようにする。preview / staging では basic auth や保守 route bypass と衝突しないことを確認する。

## 実装フェーズ

Phase 0: 現 `/admin` の位置づけ変更

- 現 `/admin` は最終版ではない。
- 使える部品は `/admin/operations` 相当へ移し、platform admin shell 配下の一機能に格下げする。
- README の「運営管理コンソール」記述は後続でこの仕様に合わせて修正する。今回の docs-only 更新では README を触らず、仕様・runbook・security matrix・issue docs を先にそろえる。

Phase 1: Platform Admin 基盤

- `PlatformOperator` / `PlatformRole` / `PlatformAuditLog` を追加
- admin サブドメインガードと `/admin` ルート保護
- admin navigation shell、検索バー、緊急状態バナー

Phase 2: 校舎ディレクトリと校舎詳細

- 全校舎横断一覧 API
- 校舎検索、フィルタ、ページング、主要メトリクス
- 校舎詳細の read-only ダッシュボード

Phase 3: 横断オペレーション

- STT、生成、配信、削除、Runpod の横断状態
- 詰まったジョブ検出
- 再実行・キャンセル操作の理由入力、監査、権限チェック

Phase 4: サポート運用

- ユーザー招待/停止/ロール確認
- Teacher App 端末 revoke
- サポートメモ、担当者、契約メタデータ
- PII アクセス時の理由入力と監査

Phase 5: 監査・運用品質

- 監査ログ検索、エクスポート
- バックアップ復元証跡、削除依頼、保持期限画面
- 負荷対策の集計テーブル、インデックス、キャッシュ

## 実装 GO 判定

この仕様は実装に進んでよい。ただし、初回実装は以下の範囲に限定する。

GO 範囲:

- 現 `/admin` ルートを platform admin の入口として作り直す。
- 既存の校舎内保守コンソールは、必要な部品だけ `/admin/operations` 相当へ移す。
- `PlatformOperator`, `PlatformRole`, `PlatformAuditLog` を追加し、校舎内 `ADMIN` と運営者権限を分離する。
- 本番では admin サブドメインを正式入口にできる構造にする。
- 初回の校舎横断管理は、既存 `Organization` を `校舎` として扱う。`CustomerAccount` / `Campus` の本格移行は後続 Issue に分ける。
- `/admin` ホームは `今日の要対応` と `校舎検索` を中心にした read-only 画面から始める。
- 校舎詳細は read-only で、概要、ユーザー件数、ジョブ状態、端末状態、最近の監査だけを出す。
- 危険操作は、理由入力・確認・監査フレームワークが入るまで露出しない。

初回ではやらない:

- 一括操作
- 代理閲覧
- 面談本文・音声の横断表示
- 契約管理の細かい編集
- 二重承認ワークフローの完全実装
- CustomerAccount / Campus の大規模データ移行
- グラフ中心の分析ダッシュボード

初回実装の合格条件:

- 校舎内 `ADMIN` だけでは `/admin` に入れない。
- 運営者は `/admin` で全校舎を検索できる。
- 運営者は校舎詳細で、校舎状態と要対応理由を本文・音声なしで把握できる。
- すべての admin API は対象校舎を明示し、`session.user.organizationId` に依存しない。
- admin write 操作を入れる場合は、必ず `PlatformAuditLog` に記録する。
- 画面は初期表示 3 領域以内、主要カード 4 つ以内、表 6 列前後を守る。
- エラーやジョブ原因は日本語で表示し、内部コードは詳細内に隠す。
